---
name: wilco-probe
description: Probe a plan, PRD, design, or codebase by scanning for issues, triaging evidence, questioning the user only where needed, and resolving conclusions into a handoff that wilco-plan can consume. Use when the user wants to inspect a design deeply, scan a codebase, inspect a repo, find the main problems in a repo, grill a plan, refine an existing plan, or converge a design before planning.
---

Preserve the full core behavior of a strong probing workflow:

- Probe the object before interrogating the user.
- Default to evidence-first triage rather than immediate questioning.
- Ask the user only when repository evidence cannot close the issue with acceptable confidence.
- Keep questioning rigorous once the workflow enters `grill`, but do not waste turns on duplicate questions or wording-only churn.
- If repository evidence and decision dominance are strong enough that one option is clearly superior, do not force a performative question. Give the recommended resolution directly.
- Treat `probe` as the umbrella verb for inspect + ask + targeted refinement. It may inspect an existing plan, grill a live question chain, or refine low-risk planning text when the target object is itself a plan.
- Treat natural requests such as `scan the repo`, `inspect the codebase`, `grill this plan`, `refine this plan`, or `how do I use wilco-probe` as valid probe entrypoints even when the user does not name the skill explicitly.
- If the user's intent could reasonably map to `probe`, `plan`, or `exec`, and repository evidence does not make one route dominant, ask a short clarification question instead of guessing.

When the discussion is about repository-native planning, prefer alignment with:

- `.wilco/prd/active/`
- `.wilco/plans/active/`
- `docs/architecture/`, if present

If the user is probing a plan or design generated by other Wilco skills, keep the feedback compatible with `wilco-plan`.
If the target object is an active plan, `wilco-probe` may refine the plan text directly for bounded low-risk clarifications, while still routing larger tracking-mode or slug changes back through `wilco-plan`.

When a question exposes an unresolved issue, keep track of which document should absorb the resolution:

- PRD-level scope, goals, non-goals, or user stories -> `.wilco/prd/active/...`
- execution sequencing, acceptance criteria, or migration slicing -> `.wilco/plans/active/...`
- durable current truth that should outlive the plan -> `docs/architecture/...`

## Input Model

Treat these as distinct input classes:

- `user steering`
  - the user's current goal, focus area, urgency, stop request, or preference about what to inspect next
- `repository evidence`
  - code, tests, docs, plans, and structural repo facts
- `current probe state`
  - issues already observed, decisions already accepted, and the current topic map

Use this priority model:

1. user steering controls direction and priority
2. repository evidence controls factual reality
3. current probe state controls continuity and next-question selection

Do not let a stale topic map override a fresh user steer.
Do not let a user factual guess silently override stronger repository evidence.

## Internal State Model

`wilco-probe` should organize its work around lightweight session-local state:

- `Intent Overlay`
  - current user steering, focus shifts, stop requests, and temporary constraints
- `Evidence Bank`
  - concrete observations from code, tests, docs, or plan artifacts
- `Issue Bank`
  - the active set of issues derived from evidence and worth triaging
- `Topic Map`
  - a derived scheduler view that groups issues into top-level themes
- `Handoff State`
  - accepted decisions, unresolved decisions, and suggested edits ready to hand to `wilco-plan`

The `Issue Bank` is the main working state.
The `Topic Map` is a derived scheduling view, not the primary truth source.
Do not persist internal probe state to repo files by default. Keep it session-local unless the user explicitly asks to write conclusions back into plan, PRD, or architecture docs.

## Issue Classes

Classify issues before deciding whether to ask the user:

- `intent`
  - scope, goals, non-goals, acceptance shape, or contract framing
  - usually writes back to PRD-level planning state or may justify `prd_backed`
- `execution`
  - sequencing, migration slices, ownership, validation order, or operational flow
  - usually writes back to the active plan
- `durable_truth`
  - architecture or policy truth that should outlive the current plan
  - usually writes back to `docs/architecture/...`

`wilco-probe` should identify the issue class and likely write-back target.
`wilco-plan` still decides whether a result remains `plan_only` or escalates to `prd_backed`.

## Question Ordering Rules

- Ask higher-impact topics before lower-impact topics.
- Prioritize questions that:
  - unblock many later questions
  - are hard to reverse later
  - can invalidate large chunks of downstream detail
- Default ordering:
  1. scope, workflow constitution, and topic model
  2. lifecycle, ownership, and routing rules
  3. invariants, upgrade gates, and stop conditions
  4. output formatting, wording, and presentational details
- When one top-level topic is sufficiently resolved, move to the next top-level topic automatically instead of staying on one topic forever.
- Do not start by polishing small format details when higher-level structure is still unsettled.
- Do not ask an unnecessary question when the recommendation is already dominant and low-risk to apply.

## Recommendation Shortcut

Use a direct recommendation instead of asking a question when all of these are true:

- one option is clearly better than the others
- repository or document evidence is already strong
- the downside of being wrong is limited
- user preference is unlikely to dominate the outcome
- the answer would not unlock materially different downstream branches

When using the shortcut:

- state the recommendation directly
- name the strongest rejected alternative if one exists
- cite the concrete evidence
- name the write-back target
- do not pretend this is still an open question

Do not use the shortcut for:

- high-irreversibility contract boundaries
- questions that primarily encode user preference or product intent
- choices with two or more genuinely live options
- decisions whose answer would materially reshape the topic map

## Global Question Selection

- Default to portfolio-style selection, not naive depth-first questioning.
- Default to evidence-first triage before user questioning.
- Maintain a lightweight question bank across the active top-level topics.
- Each top-level topic should have at most a few live candidate questions, not an unbounded backlog dump.
- Build an evidence bank and issue bank before overcommitting to any one topic.
- Build a topic map from the issue bank before overcommitting to any one topic.
- Prefer extracting the initial topic map from the object's existing structure when possible:
  - architecture and module boundaries
  - lifecycle or state-machine stages
  - interface or integration seams
  - migration and validation concerns
- Ask the user only when repository evidence cannot close the issue with acceptable confidence.
- If the object does not yet expose a strong structure, bootstrap a temporary topic map and evolve it as understanding improves.
- Rebuild or refresh the topic candidate list whenever new evidence or a new answer changes the decision tree.
- After every answered question, reassess the best next candidate across all top-level topics.
- Only stay in the current top-level topic when it still owns the highest-value next question.
- Favor topic switching when:
  - another top-level topic now has a higher-priority unresolved question
  - the current topic has already consumed multiple consecutive questions
  - the current topic is dropping from `critical` or `important` into `detail`
  - the current topic is drifting into wording, naming, or presentation refinements
- Treat the current topic as a local working context, not as a promise to exhaust that topic before reconsidering the rest of the tree.
- If the user explicitly asks to go deep on one topic, you may temporarily switch to depth mode, but portfolio mode is the default.

## Topic Coverage States

Track each top-level topic in one of these states:

- `unvisited`
  - no real question has been asked from this topic yet
- `active`
  - this topic currently owns the best next candidate question
- `deferred`
  - this topic still has unresolved candidates, but another topic currently outranks it
- `resolved`
  - this topic no longer has `critical` or `important` candidates

Do not leave topic coverage implicit. The scheduler should know which top-level topics remain unvisited, which are still active contenders, and which are effectively done.

## Issue States

Track each issue with a lightweight state:

- `observed`
  - evidence exists, but the issue has not been triaged yet
- `recommended`
  - the issue appears resolvable by direct recommendation
- `ask`
  - the issue genuinely needs user input
- `decided`
  - the issue is resolved for this pass
- `deferred`
  - the issue remains real, but another issue currently outranks it
- `dropped`
  - the issue no longer matters, was duplicate noise, or was resolved by stronger evidence

Do not confuse topic coverage with issue coverage. A topic can be visited while important issues inside it remain unresolved.

## Topic Map Evolution

Treat the topic map as a dynamic working map, not as a fixed taxonomy.

- Allow these operations as understanding evolves:
  - `add`
  - `split`
  - `merge`
  - `delete`
  - `retire`
  - `reprioritize`
- When the topic map changes, update the question bank and topic coverage state before asking the next question.
- Keep the topic map visible on demand. Show a compact snapshot when:
  - the user asks to see the current topic map
  - the user asks how topic routing or scheduling works
  - the user asks to add, delete, merge, split, or reprioritize a topic
  - the user asks how topic routing or scheduling works
  - a top-level topic is added, split, merged, retired, or reprioritized
  - the active top-level topic changes in a non-obvious way
- Do not assume the topic map should be written back to docs. Treat it as session-local working state unless the user explicitly wants it captured.

## User Topic Map Controls

If the user asks, support explicit topic-map operations during the session:

- `show`
  - display the current topic map and coverage state
- `add`
  - add a topic when the current map is missing a real concern
- `delete`
  - remove a topic when it is noise, duplicate, or no longer useful
- `split`
  - break one topic into clearer sub-topics
- `merge`
  - combine two topics when they are not meaningfully independent
- `retire`
  - remove a topic from active scheduling when it is resolved or no longer useful in this pass
- `reprioritize`
  - change ordering when the user wants to steer the probe

When the user requests a topic-map operation:

- apply it explicitly instead of treating the map as hidden internal state
- show the updated topic map if the change is non-trivial
- update candidate questions and coverage states before continuing
- do not keep stale candidate questions attached to deleted or merged topics

## Topic Map Bootstrap Heuristics

When no strong object-specific topic map exists yet, bootstrap from these heuristics:

- goals and scope
- lifecycle and state-machine behavior
- module, boundary, and interface structure
- migration and validation

These are starting heuristics, not a permanent taxonomy. Replace them with object-specific topics once the object reveals better structure.

## Question Bank

- Keep one active candidate queue per top-level topic.
- A candidate question should be concrete enough to ask next, not just a topic label.
- Prefer pruning stale or answered candidates instead of accumulating a long static list.
- When a topic is partially resolved, replace broad candidates with narrower follow-up candidates.
- When a topic no longer has `critical` or `important` candidates, deprioritize it and look elsewhere.
- The question bank should be derived from the issue bank, not invented independently from it.

## Scoring Rubric

Score topic candidates comparatively, not in isolation. Prefer the next question with the strongest combined value across:

- `impact`
  - how much of the system model changes if this answer changes
- `dependency_unlock`
  - how many later questions become clearer or unnecessary after this answer
- `irreversibility`
  - how costly it would be to answer this late instead of now
- `uncertainty`
  - how unresolved the topic currently is
- `evidence_readiness`
  - whether enough repository or document evidence exists to ask sharply now
- `duplication_penalty`
  - whether this candidate is starting to repeat accepted conclusions
- `topic_saturation_penalty`
  - whether the current top-level topic has already consumed several consecutive questions
- `unvisited_topic_bonus`
  - whether another top-level topic has not yet been entered and deserves early coverage

Use these dimensions to compare the best candidate from each top-level topic after every answer.

## Topic Scheduling Rules

- Maintain explicit top-level topic coverage state while probing.
- Give `unvisited` top-level topics an early bonus so they are not starved by one or two high-coupling topics.
- Do not ask more than `2` consecutive questions from the same top-level topic unless the user explicitly asks for depth mode.
- Once a topic falls from `critical` or `important` into `detail`, strongly prefer switching away unless no other topic has meaningful unresolved candidates.
- If two topics are close in score, prefer the less-covered topic.
- Re-rank all active top-level topic candidates after every answer before choosing the next question.
- If the current object later exposes a better topic map than the bootstrap heuristics, replace the working map instead of clinging to the original topics.

## Output Contract

Support two output modes:

- `concise` is the default mode
- `detailed` is the expanded mode
- For user-facing textual output, use the user's primary working language when it is clear from the conversation.
- If the user's preferred language is unclear, mirror the user's most recent substantive message.
- Keep commands, mode names, file paths, code literals, and machine-consumed `Probe Handoff` field names stable unless that downstream contract is explicitly changed.
- Show the current mode and available modes near the title area
- Use the compact topic header shape:
  - `<target> | <topic-path>`
  - `>` belongs to the topic path, not to the target

## Mode Contract

`wilco-probe` should support these explicit modes:

- default
  - run the default probe workflow
  - if no current issue bank exists, perform a light scan first and then route into recommendation, one live question, review, or resolve
- `scan`
  - inspect the object, build an evidence bank, derive an issue bank, derive a topic map, and report key gaps without entering a long question chain
- `grill`
  - enter a high-intensity questioning loop for the top unresolved issues after scan and triage
- `review`
  - summarize the current probe state, accepted conclusions, unresolved decisions, and likely next move without opening a new question
- `resolve`
  - stop asking new questions, collapse the current state into recommendations or a final handoff, and close the pass
- `help`
  - show available modes, topic-map controls, and common examples

When an explicit submode is invoked without a usable current issue bank:

- do not reject the mode just because `scan` was not explicitly run first
- perform a lightweight bootstrap scan first
- then continue in the requested submode using the refreshed evidence, issue, and topic state
- only refuse when even the bootstrap scan cannot establish minimally usable repository evidence

## Default Routing Rules

The default `wilco-probe` entrypoint should behave like a controller, not like a fixed questioning loop.

When the user invokes `wilco-probe` without an explicit submode:

1. build or refresh lightweight evidence and issue state first
2. classify issues into:
   - recommendation-grade
   - ask-user
   - deferred
   - dropped
3. then route using these hard rules:
   - no material issues:
     - return `review`
   - only recommendation-grade issues remain:
     - emit recommendation output directly
     - if planning handoff is clearly next, prefer `resolve`
   - one or more `ask-user` issues remain:
     - ask the single best next question directly
   - user explicitly asks for sustained questioning:
     - enter `grill`
   - user explicitly asks to close the probe:
     - enter `resolve`

Do not force `grill` as the default just because unresolved issues exist.
Use `grill` when the user wants a sustained interrogation loop or when the current pass has already committed to that mode.

### Scan Contract

`scan` should not behave like an unstructured summary. It should use a fixed diagnostic shape:

- `<target> | <topic-path>`
- `Mode: concise | available: concise, detailed`
- optional `Priority: critical|important|detail`
- optional `Intent overlay`
- `Findings`
- `Evidence`
- `Topic map snapshot`
- `High-risk gaps`
- `Likely recommendations`
- `Open questions worth asking`
- `Suggested next mode`
- optional `Suggested next target`

Use `scan` to establish the problem map, not to start a long questioning chain.
Every `Finding` should be grounded in evidence, not just speculation.
The internal `Issue Bank` still tracks the fuller issue set, but default `scan` output should emphasize user-facing findings over raw triage state.
Only expose full issue-bank style detail when the user asks for it or when unresolved issue structure materially affects the next decision.
- `scan` may legitimately suggest another narrower `scan` when the next missing input is repository evidence rather than user intent.
- Treat repeated `scan` as a controlled recursive narrowing process, not as an invitation to rescan the whole object indefinitely.
- Prefer another `scan` when a subtopic still lacks enough repository evidence to rank recommendations or ask a sharp user question.
- Stop recursive `scan` and switch modes when:
  - another narrower `scan` would not materially reduce uncertainty
  - the remaining gap is primarily user intent or preference rather than repository evidence
  - the active topic has dropped into `detail`
  - the next useful action is clearly `grill`, `review`, or `resolve`

### Grill Contract

Use `grill` when the next best action is sustained user questioning rather than more scanning.

- prefer entering `grill` after `scan` has already built an issue bank
- if `grill` is invoked without a usable current issue bank, run a lightweight bootstrap scan first instead of rejecting the request
- if a bounded, low-risk issue becomes immediately solvable during `grill`, you may briefly solve it, refresh the evidence and issue state, and then continue `grill`
- only take that shortcut when solving the issue does not require broad implementation, tracked-planning mode changes, or a fresh user preference decision
- support short answer modifiers so the user can answer and control immediate action in one line
- accept both Chinese and English short forms:
  - `推` or `rec` = choose the recommended option
  - `做` or `do` = apply immediately
  - `留` or `hold` = decide now but do not apply immediately
  - `继续` or `cont` = continue `grill` after applying or recording the answer
  - `收口` or `close` = stop questioning after this answer and route toward `review` or `resolve`
- examples:
  - `1`
  - `1 做`
  - `1 do cont`
  - `rec do`
  - `1 hold`
- focus on high-impact `ask` issues first
- keep one question active at a time
- re-rank the full topic map after every answer
- use direct recommendation instead of questioning when an issue becomes recommendation-grade
- do not re-run full scan on every turn unless new evidence or new user steering invalidates the current issue bank
- if bootstrap triage produces no real `ask-user` issue, downgrade cleanly into recommendation or `review` instead of forcing a performative questioning loop
- if an issue is solved during `grill`, mark it `decided` and do not carry it forward as an `Unresolved decision` in a later handoff
- if that solved issue still changes planning, carry only its resulting accepted rule or suggested edit into the later handoff
- when helpful, record these bounded solved items under optional `Resolved during probe` in a later handoff so downstream planning can see they were already handled

### Review Contract

`review` should show the current probe state without creating new topic-routing work. Use this shape:

- `<target> | <topic-path>`
- `Mode: concise | available: concise, detailed`
- optional `Priority: critical|important|detail`
- optional `Intent overlay`
- `Accepted decisions`
- `Unresolved decisions`
- optional `Deferred issues`
- optional `Recommended resolutions`
- optional `Topic map snapshot`
- `Suggested next mode`

`review` is a checkpoint, not a new questioning round.
If `review` is invoked without a usable current issue bank, run a lightweight bootstrap scan first and then emit the checkpoint.

### Resolve Contract

Use `resolve` when the goal is to converge the current probe state into a stable outcome instead of continuing to ask questions.

- if `resolve` is invoked without a usable current issue bank, run a lightweight bootstrap scan first instead of rejecting the request
- `resolve` does not require a prior `grill` pass
- if unresolved items are minor or recommendation-grade, emit the final summary directly
- if the result should feed planning, emit `## Probe Handoff`
- keep issue classes visible when they materially affect write-back target selection
- do not open a fresh `Question N` during `resolve`
- prefer `resolve` over an extra confirmation turn when the next useful action is clearly to hand work back to `wilco-plan`
- if a handoff is emitted, include all strong input fields explicitly
- if major `ask-user` issues still remain after bootstrap triage, keep them in `Unresolved decisions` and `Recommended resolutions` rather than opening a new question during `resolve`

### Concise Mode

Use this default shape:

- `<target> | <topic-path>`
- `Mode: concise | available: concise, detailed`
- optional `Priority: critical|important|detail`
- `Question N: <question text>`
- `Options`

Do not add the trailing response cue in concise mode.

For recommendation shortcut in concise mode:

- `<target> | <topic-path>`
- `Mode: concise | available: concise, detailed`
- optional `Priority: critical|important|detail`
- `Recommendation N: <recommended resolution>`
- `Why now`
- `Write-back target`

### Detailed Mode

Use this expanded shape:

- `<target> | <topic-path>`
- `Mode: detailed | available: concise, detailed`
- optional `Priority: critical|important|detail`
- `Question N: <question text>`
- `Options`
- `Evidence`
- `Why it matters`
- `Decision dependency`
- `Write-back target`

For recommendation shortcut in detailed mode:

- `<target> | <topic-path>`
- `Mode: detailed | available: concise, detailed`
- optional `Priority: critical|important|detail`
- `Recommendation N: <recommended resolution>`
- `Evidence`
- `Why now`
- `Counterargument`
- `Write-back target`

### Core Sections

Every detailed-mode question must include:

- `<target> | <topic-path>`
  - Use one stable target and the smallest topic path that still makes the topic obvious.
- `Mode: <current> | available: concise, detailed`
- `Question N: <question text>`
  - Keep the question text on the same line as the number.
- `Options`
  - Keep options short and mutually exclusive when possible.
  - Default to `2-4` options based on real topic complexity.
  - Put the recommended option first and mark it clearly.
  - Do not add a generic `Other` option; the user can always answer in free form.
- `Evidence`
  - Give at least one concrete basis: repository evidence, doc conflict, code observation, or already accepted decision.
- `Why it matters`
  - Explain why this question is worth asking now.
- `Decision dependency`
  - State what earlier conclusion this question depends on.
  - Keep it to `1-2` short lines.
- `Write-back target`
  - Name the file or document class that should absorb the decision.
  - If nothing should be changed yet, say so explicitly.

Every concise-mode question must include:

- `<target> | <topic-path>`
- `Mode: <current> | available: concise, detailed`
- `Question N: <question text>`
- `Options`
- optional `Priority: critical|important|detail`

Every concise-mode recommendation must include:

- `<target> | <topic-path>`
- `Mode: <current> | available: concise, detailed`
- `Recommendation N: <recommended resolution>`
- `Why now`
- `Write-back target`

### Optional Sections

Use these only when they add real value:

- `Priority: critical|important|detail`
  - Use as a lightweight label, not a full section.
  - Add it only when the sorting weight should be explicit.
- `Failure mode`
  - Use for high-risk decisions where a bad choice creates a concrete breakage pattern.
- `Confidence`
  - Use when the question or recommendation relies on incomplete evidence or meaningful inference.
- `Counterargument`
  - Use when the recommended option has a strong opposing case.
- `Not decided here`
  - Use when adjacent decisions are easy to conflate and must be kept separate.
- `Example`
  - Use when a short example removes ambiguity faster than more prose.
- `Key tradeoff`
  - Use when the real choice is a crisp tradeoff and you want to name it directly.
- `Assumptions`
  - Use when the question depends on premises that may be wrong or incomplete.
- `Next topic if accepted`
  - Use when the next step is non-obvious and you want to keep the decision tree legible.
- `Resolved so far`
  - Do not show by default.
  - Use only when the user asks for it or when a long topic needs a compact checkpoint to avoid repeated questions.
- `Topic map snapshot`
  - Use when the user asks to inspect or modify the current topic map.
  - Show current topics, coverage states, and any recent topic-map changes.

## Priority Rubric

- `critical`
  - use only when the question could change the state machine, lifecycle, distribution model, validator hard-fail behavior, runtime-closure expectations, or another system-wide contract boundary or enforcement boundary
- `important`
  - use when the question affects maintainability, ownership, migration order, or validation responsibility, but does not change a system-wide enforcement boundary or distribution/runtime contract
- `detail`
  - use for naming, presentation, local formatting, low-risk aliases, folded `scan` items, and other refinements that do not change the main decision surface
- Once a conversation drops into migration mechanics, naming, local wrappers, generated-artifact polish, or folded `scan` summaries, prefer `important` unless the answer still changes a system-wide enforcement boundary, distribution/runtime contract, or runtime model itself.

## Stop Conditions

Stop probing and move to the final summary when any of these is true:

- all `critical` and `important` topics are resolved
- only `detail`-level questions remain
- the remaining discussion is mostly wording polish
- the next question would mostly repeat an already accepted conclusion
- the next remaining item is recommendation-grade rather than a real question
- the user asks to stop
- a real blocker prevents further useful questioning

When a stop condition is met, do not ask one more confirmation question about whether to stop. Transition directly into the final summary.

## Summary Contract

At the end of a substantial probing pass, produce a final summary.

### Core Summary Sections

- `Accepted decisions`
- `Unresolved decisions`
  - Write `None` when there are none.
- `Recommended resolutions`
  - Pair these to unresolved items when unresolved items remain.
- `Suggested edits`
  - Map the accepted rules or changes to the documents that should be updated.

### Optional Summary Sections

- `Edit priority`
  - Use when multiple files or layers should be updated in a specific order.
- `Next questions`
  - Use when another probing pass is clearly warranted.
- `Not covered in this pass`
  - Use when this pass intentionally covered only part of the decision tree.

## Plan Handoff Contract

When the probe should feed directly into `wilco-plan`, emit a final handoff block.

Use this shape:

- `## Probe Handoff`
- `Target`
- optional `Slug`
- optional `Issue classes`
- `Accepted decisions`
- optional `Resolved during probe`
- `Unresolved decisions`
- `Suggested edits`
- optional `Recommended resolutions`
- optional `Edit priority`

`wilco-plan` should treat these as strong inputs:

- `Target`
- optional `Slug`
- optional `Issue classes`
- `Accepted decisions`
- optional `Resolved during probe`
- `Unresolved decisions`
- `Suggested edits`

`Resolved during probe`, `Recommended resolutions`, and `Edit priority` are soft inputs.

## Anti-Patterns

Do not:

- ask `Question N` without the actual question text on the same line
- ask a question with no concrete evidence when evidence could have been gathered first
- ask a fake question when the recommendation is already obvious
- repeat an already accepted conclusion unless the user explicitly asks for a recap
- combine multiple separable decisions into one question
- force exactly three options when the decision tree only has two real branches
- include every optional section by default and turn the probe into a long form
- keep probing once only wording polish or repeated confirmation remains
- let `agents/openai.yaml` become a second copy of the detailed format contract
- keep one top-level topic active indefinitely once its critical and important questions are already resolved
- keep drilling into one top-level topic without re-ranking the best next question across the full topic map
- starve `unvisited` top-level topics because two adjacent topics keep outranking each other on local detail
- hard-code one system's topic taxonomy as if it were the universal topic map for all grilled objects
- treat the topic map as uneditable hidden state when the user explicitly wants to inspect or change it
